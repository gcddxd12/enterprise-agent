## 1. Agent基础概念与分类

### 1.1 什么是Agent（智能体）
Agent是一个能够感知环境、做出决策并执行动作以实现目标的系统。在本项目中，Agent特指基于大语言模型的智能体，能够：
- 理解自然语言查询
- 规划解决方案
- 调用工具执行任务
- 整合结果生成最终答案

### 1.2 Agent的分类体系

#### 1.2.1 按架构分类
1. **单一Agent**（rag_agent.py）
   - 单一决策主体
   - 直接处理所有任务
   - 优点：简单直接，响应快速
   - 缺点：复杂任务处理能力有限

2. **多Agent系统**（enterprise_agent.py）
   - 多个专业Agent协作
   - 分工明确，各司其职
   - 优点：复杂任务处理能力强，模块化
   - 缺点：协调成本，延迟增加

#### 1.2.2 按工作模式分类
1. **ReAct模式Agent**
   - Reasoning + Acting 思维链+行动
   - 显式展示思考过程
   - 适合需要解释推理的场合

2. **Function Calling Agent**
   - 直接调用预定义函数
   - 隐式推理，高效执行
   - 适合工具密集场景

#### 1.2.3 按应用场景分类
1. **任务导向型Agent**（本项目的规划Agent）
   - 专注于任务拆解和分配
   - 输出结构化任务列表

2. **工具执行型Agent**（本项目的执行Agent）
   - 专注于工具调用
   - 关注执行结果收集

3. **质量验证型Agent**（本项目的验证Agent）
   - 专注于答案质量检查
   - 确保结果可信可靠

### 1.3 Agent的核心组件
1. **感知器（Perceptor）**：理解用户输入和环境状态
2. **规划器（Planner）**：制定行动方案
3. **执行器（Executor）**：调用工具执行动作
4. **存储器（Memory）**：记录历史交互和状态
5. **评估器（Evaluator）**：检查结果质量

## 3. ReAct模式深度解析

### 3.1 ReAct模式核心思想
ReAct = Reasoning（推理） + Acting（行动）
- **Reasoning**：让Agent显式思考，生成思维链
- **Acting**：基于思考结果调用工具

### 3.2 ReAct提示模板结构
```python
# 本项目中的ReAct模板（简化版）
"""
Question: {input}
Thought: {agent_scratchpad}
Action: {tool_names}
Action Input: {input}
Observation: {result}
...（可重复多次）
Thought: I now know the final answer
Final Answer: {answer}
"""
```

### 3.3 ReAct的优势与局限

#### 优势：
1. **可解释性强**：思维链展示推理过程
2. **错误容易定位**：每个步骤都可追溯
3. **适合复杂推理**：多步推理支持复杂任务
4. **人类友好**：思考过程类似人类问题解决

#### 局限：
1. **响应延迟**：需要多次LLM调用
2. **Token消耗大**：思维链占用大量上下文
3. **实现复杂**：需要精心设计提示模板

### 3.4 本项目中的ReAct实现细节

#### 工具定义：

## 4. 多Agent系统架构设计

### 4.1 多Agent协作模式

#### 4.1.1 流水线模式（本项目采用）
```
用户 → 规划Agent → 执行Agent → 验证Agent → 用户
```

**特点**：
- 线性执行，简单可靠
- 每个Agent职责单一
- 易于调试和监控
- 适合确定性的任务处理

#### 4.1.2 黑板模式（可选方案）
```
        ┌─────────┐
用户 → │ 控制Agent │ → 用户
        └─────────┘
           ↑ ↓
        ┌─────────────────┐
        │  共享状态黑板   │
        └─────────────────┘
     ↑         ↑         ↑
   Agent1   Agent2   Agent3
```

**特点**：
- 所有Agent访问共享状态
- 更灵活的协作方式
- 适合动态任务分配
- 实现复杂度高

### 4.2 本项目中的多Agent设计

#### 4.2.1 共享状态设计
```python
shared_state = {
    "user_query": "",       # 原始用户问题
    "plan": None,           # 规划Agent输出的任务列表
    "retrieved_docs": [],   # 检索到的文档
    "tool_results": {},     # 各工具执行结果
    "final_answer": None    # 验证后的最终答案
}
```

**设计考虑**：
1. **状态完整性**：记录全流程关键数据
2. **调试友好**：便于问题追踪和复现
3. **扩展性**：可添加新状态字段
4. **持久化**：为后续会话记忆做准备

#### 4.2.2 Agent间通信
1. **数据传递**：通过函数参数和返回值
2. **状态同步**：通过共享的`shared_state`字典
3. **错误处理**：每个Agent独立处理异常
4. **超时控制**：虽然没有显式实现，但可添加

### 4.3 Agent职责划分原则

#### 规划Agent职责：
1. **任务理解**：分析用户真实意图
2. **任务拆解**：将复杂问题分解为可执行单元
3. **优先级排序**：确定任务执行顺序
4. **资源分配**：为任务分配合适的工具

#### 执行Agent职责：
1. **工具调用**：按规划调用对应工具
2. **参数传递**：准备工具所需参数
3. **结果收集**：捕获工具执行结果
4. **错误处理**：处理工具调用异常

#### 验证Agent职责：
1. **质量检查**：验证答案完整性、准确性
2. **可信度评估**：检查答案是否可靠
3. **兜底处理**：当答案不可信时提供备用方案
4. **格式规范**：确保输出格式统一

## 5. 工具系统设计模式

### 5.1 工具定义规范

#### 5.1.1 装饰器模式（@tool）
```python
@tool
def knowledge_search(query: str) -> str:
    """从企业知识库中检索信息，返回答案。适用于产品使用、技术支持、销售政策等问题。"""
    qa_chain = get_qa_chain()
    if qa_chain is None:
        return "知识库服务不可用，请稍后再试。"
    result = qa_chain.invoke({"query": query})
    return result["result"]
```

**关键要素**：
1. **函数签名**：明确的输入输出类型
2. **文档字符串**：详细的工具描述和使用场景
3. **错误处理**：优雅的降级策略
4. **依赖管理**：延迟初始化资源

#### 5.1.2 Tool类模式
```python
retrieve_tool = Tool(
    name="KnowledgeBaseSearch",
    func=retrieve_and_answer,
    description="从编程知识库中检索相关信息，适合回答 Python 语法、数据结构等问题。"
)
```

**适用场景**：
- 已有现成函数需要包装为工具
- 需要更灵活的工具配置
- 动态生成工具的场景

### 5.2 工具分类体系

#### 5.2.1 检索类工具
- **知识库检索**：基于向量相似度的信息检索
- **特点**：准确性依赖向量质量，需要精心设计的嵌入模型

#### 5.2.2 查询类工具
- **工单状态查询**：查询外部系统状态
- **特点**：通常需要API集成，支持异步调用

#### 5.2.3 服务类工具
- **转人工服务**：连接人工客服
- **特点**：流程标准化，用户体验关键

#### 5.2.4 计算类工具
- **日期查询**：获取系统信息
- **特点**：确定性高，无需外部依赖

### 5.3 工具设计最佳实践

#### 5.3.1 接口设计原则
1. **单一职责**：每个工具只做一件事
2. **明确契约**：清晰的输入输出约定
3. **错误容忍**：优雅处理异常情况
4. **资源管理**：合理管理连接和资源

#### 5.3.2 性能考虑
1. **延迟初始化**：避免启动时加载所有资源
2. **结果缓存**：对相同查询缓存结果
3. **超时控制**：设置合理的执行超时
4. **并发安全**：支持多线程环境

#### 5.3.3 本项目中的工具实现亮点
```python

def get_qa_chain():
```

## 6. RAG与向量检索技术

### 6.1 RAG系统架构

#### 6.1.1 离线构建阶段
```
原始文本 → 文本清洗 → 文本分割 → 向量化 → 向量存储
```

**关键组件**：
1. **文本分割器**：`RecursiveCharacterTextSplitter`
2. **嵌入模型**：`DashScopeEmbeddings`
3. **向量数据库**：`Chroma`

#### 6.1.2 在线查询阶段
```
用户查询 → 向量化 → 相似度检索 → 上下文拼接 → LLM生成 → 最终答案
```

### 6.2 向量检索优化策略

#### 6.2.1 分块策略设计
```python
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,      # 平衡信息密度和上下文长度
    chunk_overlap=50,    # 避免信息割裂
    separators=["\n\n", "\n", "。", "，", " ", ""]  # 中文友好分隔符
)
```

**参数选择依据**：
1. **chunk_size=500**：适合大多数LLM的上下文窗口
2. **chunk_overlap=50**：确保关键信息不丢失
3. **中文分隔符**：针对中文文本特点优化

#### 6.2.2 检索参数配置
```python
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 3}  # 返回top-3最相关文档
)
```

**检索策略选择**：
1. **相似度阈值**：可设置最低相似度分数
2. **混合检索**：可结合关键词检索（BM25）
3. **重排序**：对检索结果进行二次排序

### 6.3 嵌入模型技术细节

#### 6.3.1 指令感知嵌入
```python
# 提升检索效果的格式
formatted_text = f"Instruct: {task_description}\nQuery: {query}"
```

**原理**：
- 为嵌入模型提供任务上下文
- 让模型理解"为什么"要检索
- 提升检索的相关性和准确性

#### 6.3.2 维度选择
```python
# Fireworks API中的维度配置
payload = {
    "input": text,
    "model": "fireworks/qwen3-embedding-8b",
    "dimensions": 1024  # 可选：降低维度节省存储
}
```

**维度权衡**：
1. **高维度（1536）**：表示能力更强，准确性更高
2. **低维度（1024）**：存储和计算成本更低
3. **选择标准**：根据精度需求和资源约束平衡

### 7.2 验证Agent提示策略

#### 7.2.1 基于规则的验证
```python
def validation_agent(user_query: str, final_answer: str) -> str:
    trust_keywords = ["工单", "受理", "处理中", "已完成", "今天", "年-月-日", "转人工"]
    if any(kw in final_answer for kw in trust_keywords):
        return final_answer
    if len(final_answer) < 5 or "无法确定" in final_answer:
        return "抱歉，我无法确定准确答案。建议您转人工客服。"
    return final_answer
```

**验证逻辑**：
1. **关键词验证**：检查答案是否包含可信信号
2. **长度验证**：避免过短的模糊答案
3. **负面词过滤**：识别不确定表述
4. **兜底建议**：引导用户寻求人工帮助

### 7.3 提示工程最佳实践总结

1. **明确指令**：清晰定义Agent的角色和任务
2. **结构化输出**：便于程序化处理
3. **示例引导**：提供少样本示例规范行为
4. **容错设计**：处理LLM输出的不确定性
5. **渐进细化**：复杂任务分步骤提示

```python
def run_multi_agent(user_query: str) -> dict:
    """多Agent协作主流程：体现架构设计的核心思想"""

    # 1. 规划阶段：任务拆解与结构化
    tasks = planning_agent(user_query)
    print(f"[规划] 任务清单: {tasks}")

    # 2. 执行阶段：工具调用与结果收集
    exec_results = execution_agent(tasks)
    preliminary = "\n".join(exec_results.values()) if exec_results else "无结果"
    print(f"[执行] 合并答案: {preliminary}")

    # 3. 验证阶段：质量检查与兜底处理
    final = validation_agent(user_query, preliminary)
    print(f"[验证] 最终答案: {final}")

    # 4. 结果封装：结构化返回便于前端展示
    return {
        "tasks": tasks,            # 规划结果：用于过程可视化
        "exec_results": exec_results,  # 执行结果：用于调试分析
        "final_answer": final      # 最终答案：用户可见结果
    }
```

### 10.3 规划Agent的提示工程实践
```python
def planning_agent(query: str) -> list:
    """规划Agent：体现结构化输出和容错设计"""

    prompt = ChatPromptTemplate.from_template("""
    你是一个企业客服系统的任务规划专家。请将用户的问题拆解为一系列子任务...

    # 任务类型定义（关键设计：有限的、明确的任务类型）
    - "knowledge_search: 具体查询内容"   # 从知识库检索信息
    - "ticket_query: 工单号"           # 查询工单状态
    - "escalate"                        # 转人工
    - "date_query"                      # 查询日期

    # 示例引导（关键设计：少样本学习）
    用户问"如何重置密码？" -> ["knowledge_search: 重置密码"]
    用户问"查询工单 TK-123456" -> ["ticket_query: TK-123456"]

    # 输出约束（关键设计：强制结构化输出）
    只输出 JSON 格式的任务列表，不要输出其他内容。

    用户问题：{query}
    任务列表（JSON）：
    """)

    # 执行LLM调用
    chain = prompt | get_llm()
    response = chain.invoke({"query": query}).content

    # 容错解析设计（关键设计：处理LLM输出的不确定性）
    try:
        tasks = json.loads(response)  # 首选：标准JSON解析
    except:
        # 备选：正则表达式提取
        match = re.search(r'\[.*\]', response, re.DOTALL)
        tasks = json.loads(match.group()) if match else []

    return tasks
```
